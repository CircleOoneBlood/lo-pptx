# 协作原则详解

## 核心洞察

**视觉细节用操作表达，不用语言描述。**

| 传统方式 | 协作工作流 |
|----------|-----------|
| 「把标题往左挪 20px」× 10 轮 | 直接改 → AI 看到 → 一轮搞定 |
| 「颜色再深一点，不是这个深」 | 直接选色 → AI 看到 |
| 「字号改大一点」说不清具体多大 | 直接拖动 → AI 看到精确数值 |

## 为什么 PPTX 是载体

1. **人类可直接操作**：LibreOffice/WPS 可以直接拖拽、改色、改文字
2. **结构化数据**：形状、位置、颜色、文本都是机器可读的 XML
3. **无脑对比**：通过快照对比，人类改动一览无余
4. **最终产物是图片**：PPTX 只是中间态，导出 PNG 才是交付物

## diff 的价值

`lo diff` 解决了传统工作的核心问题：

- **描述不清**：位置/颜色用语言描述有歧义，改动直接可见
- **来回试错**：AI 看到改动后可立即理解意图，无需反复确认
- **不留痕迹**：人类改动被精确记录，不被后续生成覆盖

## 改动传播链

```
人类改动 PPTX
       ↓
lo diff 对比快照
       ↓
报告变化 + 自动更新 baseline
       ↓
Agent 理解意图
       ↓
决定：接受 / 询问 / 拒绝
       ↓
如需跟进：修改代码或重新生成
       ↓
如修改了 PPTX：lo reload 热重载预览
       ↓
如满意：导出 PNG
```

## 什么改动需要告知用户

`lo diff` 报告的是**变化本身**，但**为什么改**需要人类说明。

例如：
- diff 显示：「标题文本：`初稿` → `终稿`」
- 但为什么改（客户要求 / 个人偏好 / 勘误），人类没有义务解释
- **意图不确定就问**：Agent 不确定时可以问「你是想...？」

## 不覆盖人类改动

当需要重新生成时（如 `lo init` 后调整主题）：

1. 先 `lo diff` 看人类已做的改动
2. 在代码中保留这些改动
3. 重新生成时合并进去

而不是：无脑重新生成，把人类改动覆盖掉。

## 热重载的意义

修改 PPTX 后，用 `lo reload --open` 在 LibreOffice 中打开，
人类可以立即看到效果，无需导出图片再查看。

适合：需要反复调整的场景。

## 适用场景

**适合**：
- 小红书配图（竖版 4:5）
- 社交媒体海报
- 演示文稿风格的图文
- 需要精细调整的视觉内容

**不适合**：
- 纯文字内容（直接写更快）
- 需要程序化生成的批量图片
- 复杂交互设计稿
